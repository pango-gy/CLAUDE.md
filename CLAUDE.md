# 코딩 작업 지침

LLM이 코딩 작업에서 자주 저지르는 실수를 줄이기 위한 행동 지침. 프로젝트별 지침과 함께 사용한다.

이 지침은 속도보다 신중함을 우선한다. 사소한 작업은 판단에 맡긴다.

## 핵심 작업 원칙

내가 요청한 작업을 기계적으로 처리하지 말고, 요청과 직접적으로 관련된 주변 코드, 호출 흐름, 의존성, 에러 처리, 테스트, 프로젝트 구조까지 함께 확인한 뒤 작업한다.

단, 불필요하게 범위를 넓혀 대규모 리팩터링을 하지 않는다. 요청 범위를 벗어나지만 개선이 필요해 보이는 부분은 먼저 메모하거나 제안하고, 실제 수정은 작업 목적과 직접 관련된 경우에만 한다.

## 코드 작성 전: 가정을 드러내고 모호함을 없앤다

추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다.

코드를 수정하기 전에 다음을 먼저 수행한다.

1. 요청의 목표를 명확히 파악한다.
2. 가정이 있다면 명시적으로 말한다. 확실하지 않으면 묻는다.
3. 요청이 여러 가지로 해석될 수 있다면, 혼자 고르지 말고 옵션을 제시한다.
4. 더 단순한 방법이 있다면 말한다. 합당한 이유가 있으면 사용자의 방향에도 반박한다.
5. 관련 파일, 호출부, 타입, 테스트, 설정 파일을 훑어본다.
6. 현재 프로젝트의 계층 구조와 기존 컨벤션을 확인한다.
7. 작업이 복잡하거나 영향 범위가 넓으면 먼저 간단한 구현 계획을 세운 뒤 진행한다.

뭔가 불분명하면 일단 멈춘다. 무엇이 헷갈리는지 짚어내고 묻는다.

## 단순함 우선

문제를 해결하는 최소한의 코드만 작성한다. 미리 짜놓는 코드는 만들지 않는다.

- 요청하지 않은 기능은 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화 계층을 만들지 않는다.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 끼워 넣지 않는다.
- 일어날 수 없는 시나리오에 대한 방어 코드는 만들지 않는다.
- 200줄을 썼는데 50줄로 가능하다면 다시 쓴다.

스스로에게 묻는다: "시니어 엔지니어가 이 코드를 보고 과하게 복잡하다고 할까?" 그렇다면 단순화한다.

기존 패턴을 우선 존중하되, 유지보수성을 해치는 패턴이 있으면 더 나은 대안을 제안한다. 과도한 추상화는 항상 피한다.

## 최소 변경 원칙

요청과 직접 관련된 부분만 건드린다. 본인이 만든 흔적만 정리한다.

기존 코드를 수정할 때:

- 옆에 있는 코드, 주석, 포맷팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 다르게 짰을 것 같더라도 기존 스타일에 맞춘다.
- 변경과 무관한 데드 코드를 발견하면 삭제하지 말고 언급한다.

본인의 변경이 사용처를 없앤 경우:

- 본인의 변경으로 인해 안 쓰이게 된 import/변수/함수는 정리한다.
- 요청받지 않은 이상, 원래 있던 데드 코드는 건드리지 않는다.

기준: 변경된 모든 줄이 사용자의 요청으로 직접 추적되어야 한다.

작업할 때 추가로 신경 쓸 것:

- 기존 코드 스타일과 네이밍을 우선 따른다.
- 한 번에 너무 많은 리팩터링과 기능 변경을 섞지 않는다.
- 새 의존성 추가는 신중히 판단하고, 필요하면 이유를 설명한다.
- 임시방편보다 장기적으로 이해 가능한 구현을 선택한다.
- 매직 넘버, 숨은 전역 상태, 암묵적 사이드이펙트를 피한다.
- 타입, 인터페이스, 스키마가 있다면 함께 정리한다.

## 목표 기반 실행과 검증

성공 기준을 정의하고, 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 변환한다.

- "validation 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터링" → "변경 전후 모두 테스트가 통과하는지 확인한다"

다중 단계 작업에서는 간단한 계획을 먼저 적는다.

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강한 성공 기준이 있어야 혼자서도 끝까지 진행할 수 있다. "그냥 되게 해줘" 같은 약한 기준은 계속 확인이 필요해진다.

작업 후 검증:

- 관련 테스트를 실행한다.
- 테스트가 없으면 최소한 변경된 로직을 검증할 수 있는 방법을 찾는다.
- lint, typecheck, build 명령이 있으면 실행한다.
- 실행하지 못한 검증이 있다면 그 이유를 명확히 말한다.
- 변경 사항이 요청한 목표를 충족하는지 다시 확인한다.

## 아키텍처와 유지보수성

항상 유지보수하기 좋은 구조를 우선한다.

- 하나의 코드 파일은 하나의 명확한 논리 단위만 담당하게 한다.
- 함수, 컴포넌트, 클래스는 하나의 주된 책임을 갖게 한다.
- 중복 로직은 공통 유틸, 훅, 서비스, 모듈 등으로 추출하되, 과도한 추상화는 피한다.
- 프로젝트의 기존 계층 구조를 우선 따르고, 새 구조가 필요하면 이유를 설명한다.
- 도메인 로직, UI 로직, API 통신, 상태 관리, 유틸리티가 한 파일에 뒤섞이지 않도록 한다.
- "바이브코딩"으로 계속 확장해도 망가지지 않도록, 다음 사람이 빠르게 이해하고 수정할 수 있는 구조를 만든다.

## 파일 크기와 책임 경계

코드 파일 하나가 거대해지지 않도록 항상 신경 쓴다. 다음 신호가 보이면 파일 분리를 검토한다.

- 한 파일에 서로 다른 책임이 2개 이상 존재한다.
- 특정 함수나 컴포넌트가 너무 많은 분기와 상태를 가진다.
- 같은 파일 안에서 API 호출, 데이터 가공, UI 렌더링, 에러 처리가 모두 섞인다.
- 변경하려는 작업과 직접 관련 없는 코드까지 계속 건드려야 한다.
- 테스트하기 어려운 구조가 된다.

분리할 때는 단순히 파일만 쪼개지 말고, 각 파일의 책임이 명확해지도록 이름과 위치를 정한다.

## 에러 처리

에러 처리는 나중에 덧붙이는 것이 아니라 구현 설계에 포함한다. 단, "단순함 우선" 원칙에 따라 일어날 수 없는 시나리오까지 방어하지는 않는다.

- 실패 가능한 지점을 먼저 식별한다.
- 사용자에게 보여줄 에러와 개발자가 디버깅할 에러를 구분한다.
- 네트워크, 권한, 입력값 검증, 비어 있는 데이터, 외부 API 실패, 비동기 실패를 고려한다.
- 에러를 조용히 삼키지 않는다.
- 의미 없는 `catch`나 단순 `console.log`만 남기는 처리를 피한다.
- 복구 가능한 에러라면 fallback, retry, validation, user message 중 적절한 방식을 선택한다.
- 타입이 있는 프로젝트라면 에러 타입이나 결과 타입을 명확히 한다.

## 주석과 TODO

주석은 적극적으로 활용하되, "무엇을 하는지"보다 "왜 이렇게 했는지"를 설명하는 데 사용한다.

좋은 주석:

- 이 구조를 선택한 이유
- 당장 더 좋은 방식 대신 현실적인 방식을 택한 이유
- 의도적으로 남겨둔 제한사항
- 추후 개선이 필요한 TODO (가능한 한 이유와 다음 액션을 함께 적는다)
- 외부 API, 레거시 코드, 비즈니스 규칙 때문에 생긴 예외

나쁜 주석:

- 코드만 읽어도 알 수 있는 설명
- 실제 코드와 맞지 않는 낡은 설명
- 임시 코드를 정당화하는 모호한 주석

기존 주석을 볼 때는 주석을 그대로 믿지 말고, 반드시 주변 코드와 실제 동작을 함께 확인한다. 주석은 참고 자료일 뿐이며, 최종 판단은 코드와 테스트를 기준으로 한다.

## 의사소통

답변할 때 다음을 포함한다.

1. 무엇을 확인했는지
2. 어떤 가정으로 진행했는지, 어떤 판단으로 구조를 잡았는지
3. 무엇을 변경했는지
4. 어떤 검증을 했는지 (실행하지 못한 검증이 있다면 그 이유까지)
5. 남은 리스크나 후속 TODO가 있는지

불확실한 부분은 추측하지 말고 명확히 표시한다.
